# Dispatcher system lift implementation

**Goal**: Refactor the architecture visualization so that concrete implementation packages (e.g., container-dispatcher-server) are omitted from the system diagram. Instead, lift transitive uses relations from those implementation nodes up to the abstract dispatcher architecture node. This ensures the diagram reflects the logical architecture without clutter from concrete packages.

## Decisions taken
- **Keep abstract dispatcher node**: The system view will display the dispatcher as an architectural component.
- **Lift transitive uses**: Any uses edges originating from implementation packages that are reachable from the dispatcher will be re‑wired directly to the dispatcher node.
- **Exclude concrete implementation packages**: Packages such as container-dispatcher-server will not appear as separate nodes in the diagram.
- **Preserve original relations**: Direct uses relations that do not involve the dispatcher hierarchy remain unchanged.

## Rejected alternatives
- **Show all implementation packages**: Decided against because it created a noisy diagram that hid the high‑level architecture.
- **Collapse all nodes into a single package**: Rejected as it would lose important dependency information for other subsystems.

## Implementation details (abstract)
- Parse the system graph generated by the existing tooling.
- Identify the subtree rooted at the dispatcher node.
- Walk the subtree to collect all outgoing uses edges.
- For each collected edge, create a new edge from the dispatcher node to the target of the original edge.
- Remove nodes that belong to the concrete implementation packages from the final graph output.
- Ensure the modified graph is still valid and can be rendered by the visualization pipeline.

## Acceptance criteria
- The generated system diagram no longer contains concrete dispatcher implementation packages.
- All transitive dependencies originally reachable through those packages are now directly connected to the dispatcher node.
- The build (npm run build) passes without errors.
- End‑to‑end tests (npm run e2e in packages/example/local-docker) pass.
